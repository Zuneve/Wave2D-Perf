# Wave2D-Perf.

Исследовательский стенд для изучения **пределов производительности современных CPU**
на примере численного решения 2D уравнения Шрёдингера.

Цель не столько в физической точности, сколько в том, чтобы зафиксировать
поведение железа на типичном stencil-вычислении и найти узкие места
через профилирование и анализ.

В рамках проекта мы будем развивать вычисления: начнем с наивной версии (с использованием **std::complex<float>**) и будем двигаться к более оптимизированным вариантам.

Отслеживать прогресс и ценность проекта будем при помощи бенчмарков и подробного профилирования вычислительного процесса.

Текущее состояние:
- `euler` — наивный baseline на `std::complex<float>`.
- `cn-adi` — основной solver, переписанный на SoA-представление (`real[]`, `imag[]`, `potential[]`) без `std::complex` в горячем цикле.

Что уже оптимизировано в текущем `cn-adi`:
- коэффициенты Cayley для неизменного потенциала предвычисляются один раз перед временным циклом;
- арифметика в прогонке переписана под специальные формы `1 ± iβ` и `iα`, чтобы убрать часть лишних общих complex-операций;
- `x`- и `y`-проходы распараллелены через `std::thread` по независимым строкам/столбцам с отдельными Thomas-буферами на поток;
- в `y`-проходе добавлена SIMD-обработка пакетов из 4 соседних столбцов как 4 независимых прогонок в одном векторе;
- SoA layout и padding по строкам сохранены как база для дальнейшей SIMD-ветки.

Практический эффект текущих оптимизаций:
- микроправки в арифметике сами по себе дают лишь небольшой прирост;
- основной выигрыш сейчас приходит от многопоточности:
  на `256x256`, `1000` шагов теперь примерно `32.4 -> 82.0 MLUPS` (`threads=1 -> 8`);
  на `512x512`, `300` шагов теперь примерно `43.5 -> 136.7 MLUPS` (`threads=1 -> 8`);
- SIMD в `y`-проходе дал дополнительный прирост поверх уже работающей многопоточности.

Ограничение текущей версии:
- главный bottleneck по-прежнему в tridiagonal solve и особенно в memory-access pattern `x`-прохода, поэтому следующий заметный шаг вверх всё ещё связан с дальнейшей SIMD-оптимизацией и балансировкой доступа к памяти.

## Физика задачи

Решаем нестационарное уравнение Шрёдингера (ℏ = 1):

```
i ∂ψ/∂t = -(1/2m) ∇²ψ + V(x,y) ψ
```

По умолчанию используем более корректную для длительной эволюции схему
**CN-ADI (Crank-Nicolson / Alternating Direction Implicit)**.
Она заметно лучше подходит для уравнения Шрёдингера, чем явный Эйлер,
потому что гораздо лучше сохраняет норму волновой функции и не разваливает решение так быстро.

Явный Эйлер оставлен в проекте только как **наивный baseline** для сравнения производительности
и демонстрации численной деградации.

Для baseline-интегратора:

```
∂ψ/∂t = i/(2m) ∇²ψ - i·V·ψ

∇²ψ[i,j] ≈ (ψ[i-1,j] + ψ[i+1,j] - 2ψ[i,j]) / dx²
           + (ψ[i,j-1] + ψ[i,j+1] - 2ψ[i,j]) / dy²
```

Граничное условие: Дирихле (ψ = 0 на границах).
Начальное состояние: гауссов волновой пакет + гармонический потенциал + двойная щель.

## Сборка

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j

# или без агрессивных оптимизаций (для анализа кода):
cmake -B build_debug -DCMAKE_BUILD_TYPE=Debug -DWAVE2D_FAST_MATH=OFF -DWAVE2D_NATIVE_ARCH=OFF
cmake --build build_debug -j
```

## Запуск

```bash
# Одиночный замер физически более разумной схемы по умолчанию
./build/wave2d --nx 1024 --ny 1024 --steps 400

# Тот же solver, но с явным управлением числом потоков
./build/wave2d --nx 1024 --ny 1024 --steps 400 --threads 8

# Наивный baseline на явном Эйлере
./build/wave2d --integrator euler --nx 1024 --ny 1024 --steps 400

# Sweep по размерам сетки → CSV
./build/wave2d --sweep --steps 200
python3 tools/plot_benchmark.py benchmark.csv

# Дамп симуляции → анимация
./build/wave2d --nx 512 --ny 512 --steps 25000 --dump-every 250 --output sim.bin

python3 tools/animate_simulation.py sim.bin -o wave.mp4  # сохранить видео
```
